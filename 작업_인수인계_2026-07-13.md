## 1. 프로젝트 개요

- **서비스**: 모모시티(momocity) — "카테고리를 학습하고 나만의 도시를 키우는" 게이미피케이션 학습 플랫폼
- **팀**: MoMoFighters, 담당: FE
- **스택**: Next.js 16(App Router) + React 19 + TypeScript, Tailwind CSS v4(CSS-first, `tailwind.config.*` 없음), shadcn/radix-ui, TanStack Query, axios, socket.io-client + `@stomp/stompjs`(실시간), FullCalendar, `next/image`, `next/font/google`
- **인증/데이터 패턴**: Server Actions(`"use server"`), 레이아웃 단에서 쿠키의 JWT를 디코드해 접근 제어, `fetchWithAuth`(`src/lib/api.ts`)가 `cookies()`에서 accessToken을 꺼내 `Authorization: Bearer`를 자동으로 붙여줌 → 이 함수를 쓰는 라우트는 전부 동적 렌더링됨
- **로컬 환경**: 프론트 `localhost:4444`, 백엔드(스프링) `localhost:8080`(`.env`의 `NEXT_PUBLIC_API_BASE_URL`)

## 2. 작업 규칙 (필독 — 프로젝트 설정에 등록된 규칙)

1. **코드를 직접 수정하지 않는다.** 항상 수정하려는 전체 코드를 코드블럭으로 먼저 보여주고 설명한 뒤, 사용자가 "적용해줘" 등으로 확정하면 그때 실제로 반영한다.
2. 코드블럭 제안 시엔 현재 코드와 변경되는 부분에 주석을 달아서 보여주고, 실제로 적용할 때는 그 설명용 주석은 제거하고 깔끔한 코드로 반영한다.
3. 코드는 항상 **초기 렌더링 개선 / UX 개선**을 고려한 최적화된 코드여야 하며, 작업이 끝나면 어떤 최적화가 반영됐는지 설명한다.
4. 최적화 관련 질문을 받으면 **공식 문서(Next.js 공식 사이트 등)를 검색해서** 근거 기반으로 답한다.
5. 한글 문자열이 깨져 보이면 먼저 읽는 과정(내 쪽) 문제로 보고 UTF-8로 다시 읽어본다. 그래도 안 되면 그때 사용자에게 알린다.
6. 디자인을 새로 하거나 전체적으로 수정할 때, 또는 사용자가 반응형을 요청할 때는 **모바일 환경을 특히 세심하게** 고려한다.

### 환경 관련 참고사항 (버그 아님, 특성)
- 연결된 폴더(`C:\module3\FRONT-Fighters`)는 Linux 샌드박스 bash에도 마운트되어 있는데, **텍스트 파일을 File 도구(Read/Edit/Write)로 수정한 직후 bash로 읽으면 오래된/깨진 내용**(trailing NUL, 잘림 등)이 보일 때가 있음. 이건 bash 마운트 쪽의 지연/캐시 문제이고, **File 도구(Read)의 결과가 항상 진실**이므로 그걸 기준으로 판단할 것. 반면 이미지 같은 바이너리 파일은 bash로 써도 정상 반영됨.

### 3-7. 코드 레벨 최적화 (sizes / priority / optimizePackageImports / fetch 캐싱)
공식 문서(Next.js `Image` 컴포넌트, `fetch`, `optimizePackageImports`, 캐싱 가이드) 기준으로 확인 후 진행:

- **`sizes` prop 감사**: `BuildingItem`(imageSizes prop), `BusStation`, `PostBoard`, `Phone`, `GuestHeroSection`은 이미 잘 지정돼 있었음. 유일하게 누락된 `src/components/city/CityCanvas.tsx`(도시 배경, `fill`+`priority`)에 **`sizes="100vw"` 추가 완료** — 반응형 srcset(640~3840w)이 생성되도록 함
- **`priority` deprecated 발견**: Next 16부터 `priority`가 deprecated, `preload`로 대체 권장(공식 문서 확인). 코드에 `priority`를 쓰는 파일 15개 발견(`CityCanvas`, `StudentHeader`, `TeacherHeader`, `AdminHeader`, `GuestHeader`, `LoginForm`, `AuthPageShell`, `not-found`, `error`, `forbidden`, `BuildingItem`, `Phone`, `StudentCityBuildings`, `GuestLectureDetailHero`, 친구 도시 페이지) — **아직 미적용**, 필요 시 `preload`로 일괄 전환 가능(작동은 계속 되므로 급하지 않음)
- **`next.config.ts`**: `experimental.optimizePackageImports`에 `"radix-ui"` 추가 완료 (기본 최적화 목록엔 lucide-react/date-fns/recharts만 있고 radix-ui는 없음)
- **fetch 캐싱 전략 재정리 및 적용**:
  - 핵심 원리: 라우트가 `cookies()`로 인해 동적 렌더링되는 것과, 개별 `fetch`가 Data Cache에 캐싱되는 것은 서로 다른 축. 명시적 `cache`/`next.revalidate` 옵션은 라우트의 동적 여부와 무관하게 항상 존중됨(공식 문서 확인)
  - 단, `fetchWithAuth`가 사용자별 JWT를 헤더에 실어 보내므로 캐시 키가 사용자마다 달라짐 → "전 사용자 공유 캐시"는 아니고 "같은 사용자의 반복 요청"에 대해서만 캐시 히트
  - 잔디(오늘 날짜만, 최대 레벨 4까지, 그 이후/이전 날짜는 불변)와 건물(레벨 시스템 없음, 한 번 생성되면 불변)은 **"언제 바뀌는지 정확히 아는" 데이터** → 시간 기반 `revalidate` 폴링보다 **이벤트 기반 무효화(`revalidatePath`)** 가 더 정확하고 효율적
  - 실제 코드 확인 결과, `src/features/lecture/action.ts`의 `updateVideoProgressAction`/`updateVideoProgressByExitAction`(영상 시청 진척도 저장 = 잔디·건물이 바뀔 수 있는 유일한 시점)이 이미 `revalidatePath`를 여러 경로에 걸고 있었는데, 정작 **`/student`(도시 메인 페이지)가 빠져 있었음** → 이게 `getMyBuildings`에 `no-store`를 걸어둔 이유로 추정됨
  - **적용 완료**:
    - `src/features/lecture/action.ts`의 `revalidateLectureProgressPaths`에 `revalidatePath('/student')` 추가
    - `src/app/services/city/service.ts`의 `getMyBuildings`를 `cache: "no-store"` → `next: { revalidate: false }`(무기한 캐시, 진척도 저장 시점에 정확히 무효화되므로 안전)로 변경
  - `getMyStreak`/`getFriendStreak`의 `revalidate: 60`은 그대로 둠

### 3-8. 서비스 함수 전체에 렌더링 전략(캐시 옵션) 명시 적용 (완료)
관리자 도메인은 "한 명만 접속"이라는 특성상 시간 기반 캐시보다 **SSR(`no-store`)로 항상 최신화**하는 게 맞다는 방향으로 최종 정리해서 전 서비스 함수에 명시적으로 반영함.

- **`no-store`(SSR) 적용** — 관리자 도메인:
  - `report/service.ts`: `getReports`
  - `admin-dashboard/service.ts`: `getMonthlyState`, `getMonthlySubState`, `getDashboardSummary`
  - `access-log/service.ts`: `getAccessLogs`
  - `user/service.ts`: `getUsers`, `getUserById`, `getPendingTeachers`, `getPendingTeacherById`
  - (`getReportList`, `getReportDetail`은 이미 `no-store`였음 — 유지)
- **`no-store` 적용** — 개인화·실시간 데이터:
  - `notification/service.ts`: 알림 카운트/목록 3개 함수
  - `point/service.ts`: `getProfileOrderListService`(내부 storeResponse 조회 포함), `getPointHistoryListService`
  - `lecture/service.ts`: `getProgressByCategory`, `getLatestChapterInfo`, `getVideoPlayer`
  - `community/service.ts`: `getCommunityPostDetailService`, `getCommunityPostRecommendationsService`, `getCommunityPostCommentsService`, `getCommunityPostRepliesService`
  - `phone/calendar/service.ts`: `getMonthlyCalendar`, `getTodoList`
- **`revalidate: 60`(ISR) 적용** — 다수 사용자 대상, 자주 안 바뀌는 목록/상세:
  - `lecture/service.ts`: `getLecturesWithAuth`, `getLectureById`, `getReviewsByLectureId`, `getLectureMeta`
  - `point/service.ts`: `getPointStoreItemsService`
  - `city/service.ts`: `getFriendBuildings`
- **`force-cache` + `revalidateTag`(태그 기반 무효화) 적용** — "관리자가 등록/수정/삭제할 때만 바뀌고, 그 외엔 절대 안 바뀌는" 공지사항:
  - `notice/service.ts`: `getNotices`, `getNoticeById`를 `cache: "force-cache"` + `next: { tags: ["notices"] }`로 변경
  - `notice/action.ts`: 등록/수정/삭제/고정·고정해제 4개 액션 전부에 `revalidateTag("notices")` 추가(기존 `revalidatePath`는 유지) → 나중에 비로그인 사용자가 보는 화면이 생겨도 등록/수정/삭제 시점에 정확히 캐시가 갱신됨
